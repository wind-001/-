## 1. 推荐结果解释（为什么推荐给我）

### 目标

在首页、排行榜NCF推荐榜、详情页底部等所有推荐列表中，为每部电影附加一句简短的解释文案。

### 实现方案

根据当前使用的推荐策略，在后端生成解释字符串，随电影数据一起传到模板。

#### 策略对应解释

- **热门推荐** (`get_hot_movies`)：
  解释文案固定为 `"大家都在看"` 或 `"近期热门"`。
- **偏好推荐** (`get_preference_based_recommend`)：
  读取用户在注册时选择的偏好类型（存于 `UserProfile` 的某个字段，假设为 `preferred_genres`，JSON 数组）。
  解释文案：`"因为你的喜好类型：科幻、动作"`（展示前2个类型即可）。
- **NCF 协同过滤** (`get_ncf_recommendations`)：
  核心是找到**用户最近操作中最相似的已看电影**，关联到当前推荐。
  **具体步骤**：
  1. 从 Redis db3 取出该用户最近100条行为记录（已有）。
  2. 用已有权重和时间衰减计算出每条记录的得分，取**得分最高的一条记录**对应的电影 `movie_id`。
  3. 查询该电影标题 `source_title`。
  4. 计算该 `source_movie` 的 item embedding 与当前推荐电影 embedding 的余弦相似度（都已存在），可作为一个内部旁证，但不强制展示。
  5. 解释文案生成：`"因为你喜欢《星际穿越》"` 或 `"和你看过的《星际穿越》相似"`。

#### 缓存与性能

- 每部推荐电影的 `reason` 字符串可以在循环内直接生成，不涉及大量额外计算。
- 如果需要，可以在生成整个推荐列表时顺带产出 `reason` 字段，放入列表的字典中。

#### 前端展示

在电影卡片或列表项中增加一行 `<span class="rec-reason">{{ movie.reason }}</span>`，用灰色小字显示。

### 涉及的代码位置

- `recommend/views.py` 中各推荐函数返回的数据结构中添加 `'reason'` 字段。
- 模板中调用 `{{ movie.reason }}`。

## 4. 用户反馈循环（即时更新用户向量）

### 目标

让推荐引擎在用户产生新行为（评分、收藏、搜索后）立即感知兴趣变化，**避免每次都重新遍历 100 条历史记录**，从而加快下一次推荐响应，并让用户感到系统“实时响应”。

### 设计思路

目前 `get_ncf_recommendations()` 的逻辑是：每次推荐时从 Redis 取出最近 100 条行为记录 → 加权 + 时间衰减 → 合成一个用户向量 → 与所有电影 embedding 计算余弦相似度。
**改进**：在 Redis 中直接维护一个 **用户向量快照** `user_vec:{user_id}`（32 维 float 列表），行为发生时**增量更新**该向量，推荐时直接使用。

#### 向量更新算法

当用户对电影 `m` 进行一次操作时：

1. 获取当前用户向量 `V`（如果不存在则全零向量）。
2. 获取电影 `m` 的 item embedding（来自 NCF 模型，可缓存为字典 `movie_id -> embedding`）。
3. 计算行为权重 `w`（点击=0.3，收藏=0.7，评分=1.0，搜索收藏同上）。
4. 时间衰减处理：为了简化，可以在每次更新时对旧向量乘以一个衰减因子 `γ = exp(-0.01 × Δt)`，其中 `Δt` 是距上次更新秒数（需要存储 `last_update` 时间戳）。
   即：`V_new = V_old * γ + w * E_m`。
5. 将新向量 `V_new` 及当前时间戳存回 Redis。

**优点**：推荐时只需取 `user_vec` 与所有电影嵌入矩阵做一次余弦相似度计算，性能提升，且新行为立即影响后续推荐。

#### 降级与初始化

- 如果 `user_vec` 不存在（如新用户），则退化为读取历史记录重新计算，并顺势生成 `user_vec`。
- 模型重新训练后，item embedding 会变化，可设置一个全局版本号，检测到版本变化时批量清空所有 `user_vec`，下次推荐时重新计算。

### 涉及的修改

- **行为追踪处**：在 `click`、`collect`、评分提交的视图中，在写入 Redis 行为记录之后，调用一个 `update_user_vector(user_id, movie_id, weight)` 函数。
- **新增工具函数**：`update_user_vector`，从 Redis 读取/写入 `user_vec:{uid}` 和 `user_vec_time:{uid}`；需要读取电影 embedding（可预加载的内存字典或缓存）。
- **推荐函数**：`get_ncf_recommendations` 中增加分支——优先取 `user_vec`，若不存在再回退到全量历史计算。
- **模型重训命令**：增加清空所有 `user_vec:*` 键的步骤。

### 答辩亮点

“采用增量式用户向量更新，通过指数衰减与实时注入新行为，实现近实时的兴趣跟踪，避免每次推荐的全量计算。”

## 5. A/B 测试框架

### 目标

设计一个可对比推荐策略的实验系统，证明新老策略的优劣，体现**在线评估**和**数据驱动决策**能力。

### 实现方案

#### 5.1 用户分组

- 在 `UserProfile` 模型中添加字段：
  `rec_group = models.CharField(max_length=1, default='A')` # 'A' 或 'B'
- 分组时机：
  - **注册时**随机分配（50%/50% 概率）。
  - 已注册老用户，可在第一次登录新系统时检测分组为空，随机分配并保存。
- 为便于分析，还可记录 `group_assigned_time`，但非必须。

#### 5.2 策略分流

在所有推荐接口（首页、分类、排行榜 NCF 推荐）中，根据 `request.user.profile.rec_group` 切换推荐逻辑：

- A 组：原来的推荐策略（如上文的三阶段）。
- B 组：新的实验策略，例如加入多样性打散、调整权重、或使用新的混合推荐比例。
  为了让实验有意义，B 组必须有一个**可对比的差异点**（比如同是 NCF 但关闭探索模式，或使用不同的时间衰减系数等）。如果暂时没有新策略，也可以让 B 组保持原样，框架先搭好，答辩时能讲清楚流程即可。

#### 5.3 指标收集

记录每一次**点击**和**收藏**行为，并带上分组信息与推荐场景，以便离线计算点击率（CTR）或收藏率。

**新增数据表** `rec_experiment_log` 可选，或直接将分组信息附加到现有行为记录中：

#### 5.4 效果统计与展示

在个人中心增加一个“实验数据”卡片（仅管理员或所有用户可见当前自己的组别和统计），展示：

- 自己所属实验组（A 或 B）。
- 最近 7 天自己的点击次数、收藏次数（可来自该表）。
- 系统级别的两组总指标对比（如 A 组 vs B 组的 CTR），通过一个简单的柱状图呈现。

**系统级统计计算**：

- 在个人中心视图（或独立的管理视图）中，查询 `RecEvent` 表，按 `rec_group` 分组，统计总曝光（需要记录展示，稍复杂），或简化用**点击次数/推荐请求次数**近似。
- 简单做法：记录每次推荐请求的返回电影数作为曝光（`RecExposure` 表），然后统计点击率。但为了缩短实现，**可以先只做“收藏率”对比**，因为收藏行为明确。
  CTR 的计算可简化为：`在推荐发生后的 5 分钟内，用户点击了推荐列表中的电影，记作一次点击通过`，可以用 Redis 临时记录。
- 对于毕设，可在后台写一个管理命令计算各组的**人均收藏数**或**平均点击次数**，存入 Redis 缓存，然后在个人中心展示两组数字对比。

**前端图表**：

- 使用 Chart.js CDN 引入，绘制一个简单的柱状图：X 轴显示“A 组”“B 组”，Y 轴显示“人均收藏数”或“点击通过率”。
- 数据通过一个轻量 Ajax 接口获取 JSON：`{"A": 3.2, "B": 4.1}`。

### 针对你的项目可以怎么对比？

一个最有意义的实验：**对比原始热门推荐 vs NCF 推荐在过渡期用户上的效果**，或者对比有探索模式 vs 无探索模式的推荐。你可以将 B 组设置为：在第三阶段始终关闭探索模式，保留纯 NCF 结果；A 组使用原有的 70%匹配+30%探索。然后观察两组用户的点击率或收藏率差异，写成实验报告。
